# Spec Refine

**Specification (optional):** $ARGUMENTS

---

## Workflow Instructions

This command runs validation on a specification, then interactively resolves any issues, questions, or concerns. Run multiple times until the spec is ready for implementation.

### Step 1: Locate Specification

**If a path was provided:** Use that path directly.

**If no path was provided:** Find the most recently modified `02-specification.md` file in `doc/specs/`:
1. Search for all `02-specification.md` files under `doc/specs/`
2. Select the one with the most recent modification time
3. Inform the user which file was auto-selected:
   ```
   Auto-selected: doc/specs/{slug}/02-specification.md (modified {time})
   ```

Extract the feature slug from the path for later use.

### Step 2: Check for Existing Unresolved Feedback

Before running validation, check if the spec already has a "## Validation Feedback" section with unresolved items (items NOT struck through with `~~`).

**If unresolved items exist:** Skip to Step 4 (Present Issues for Resolution) using the existing items.

**If no unresolved items:** Proceed to Step 3 to run fresh validation.

### Step 3: Run Validation and Persist Results

Use and follow the `spec-validate` skill exactly as written to analyze the specification at `doc/specs/{slug}/02-specification.md`.

Capture and parse the validation results for:
- **Critical Gaps** - Must-fix issues blocking implementation
- **Missing Details** - Areas needing clarification
- **Open Questions** - Unresolved decisions in the spec
- **Overengineering Concerns** - Features to cut or simplify
- **Risk Areas** - Potential implementation challenges

**If validation shows Ready status:**

Check if there are any "Recommendations" or "Missing Details (Minor)" items.

**If no recommendations at all:**
```
Specification Ready ✅

The specification has passed validation and is ready for implementation.

Next Step: Run /spec:decompose doc/specs/{slug}/02-specification.md
```
Exit the command.

**If recommendations exist, present them using AskUserQuestion:**

```
Specification Ready ✅ (with {N} minor recommendations)

The specification is implementation-ready, but validation identified optional improvements.
```

For each recommendation, use AskUserQuestion:
```
AskUserQuestion:
  questions:
    - header: "Rec {N}"
      question: "{Recommendation description}. Apply this improvement?"
      multiSelect: false
      options:
        - label: "Apply"
          description: "Update the spec with this improvement"
        - label: "Skip"
          description: "Proceed without this change"
```

If user selects "Apply": Update the spec accordingly, then continue to next recommendation.
If user selects "Skip": Log as "Acknowledged, not applied" and continue.

After processing all recommendations, proceed to Step 6 (Summary).

**If actionable items found:** Append or update a "## Validation Feedback" section at the end of the spec document:

```markdown
## Validation Feedback

*Generated by /spec:refine on {date}*

### Critical Gaps
1. {Gap description}
2. {Gap description}

### Missing Details
1. {Detail needed}
2. {Detail needed}

### Open Questions
1. {Question}
2. {Question}

### Overengineering Concerns
1. {Concern} - Recommendation: {recommendation}
2. {Concern} - Recommendation: {recommendation}

### Risk Areas
1. {Risk description}
```

This persists the validation results so they're not lost if the session is interrupted.

### Step 4: Present Issues for Resolution

For each actionable item from the validation, use **AskUserQuestion tool** to gather decisions:

1. **Display the issue** in your message before asking:
   ```
   Issue {current} of {total}: {Category}
   {Issue description from validation}
   ```

2. **Use AskUserQuestion tool** based on issue type:

   **For Critical Gaps / Missing Details:**
   ```
   AskUserQuestion:
     questions:
       - header: "Gap {N}/{total}"
         question: "{Describe the gap}. How should we address this?"
         multiSelect: false
         options:
           - label: "Provide details"
             description: "I'll provide the missing information"
           - label: "Not needed"
             description: "This detail isn't necessary for implementation"
           - label: "Defer"
             description: "Address in a future iteration"
   ```

   **For Overengineering Concerns:**
   ```
   AskUserQuestion:
     questions:
       - header: "Simplify?"
         question: "{Concern}. Recommendation: {recommendation}. How to proceed?"
         multiSelect: false
         options:
           - label: "Cut it (Recommended)"
             description: "Remove from spec entirely"
           - label: "Simplify"
             description: "Reduce scope/complexity"
           - label: "Keep as-is"
             description: "Disagree with assessment"
           - label: "Defer"
             description: "Move to future work"
   ```

   **For Risk Areas:**
   ```
   AskUserQuestion:
     questions:
       - header: "Risk"
         question: "{Risk description}. How should we handle it?"
         multiSelect: false
         options:
           - label: "Add mitigation"
             description: "I'll provide a mitigation strategy"
           - label: "Accept risk"
             description: "Acknowledge and document in spec"
           - label: "Reduce scope"
             description: "Cut the risky portion"
   ```

3. **If user selects "Provide details" or "Add mitigation"**, follow up with another AskUserQuestion or request text input.

4. **Record the decision** for each issue

### Step 5: Update Specification

For each resolved issue, update the specification document:

1. **Re-read the spec** to get current content

2. **Apply updates based on decisions:**

   - **Missing details resolved:** Add the new information to the appropriate section
   - **Questions answered:** Update the Open Questions section with strikethrough format:
     ```markdown
     ~~1. Original question~~ (RESOLVED)
     **Answer:** User's decision
     ```
   - **Features cut:** Remove from relevant sections, add to "Out of Scope" or "Deferred Work"
   - **Risks mitigated:** Add mitigation strategy to relevant section
   - **Simplifications:** Update the affected sections with simpler approach

3. **Mark resolved in Validation Feedback section:**

   After resolving each item, update it in the Validation Feedback section with strikethrough:
   ```markdown
   ~~1. {Original issue}~~ (RESOLVED)
   **Resolution:** {How it was addressed}
   ```

   This ensures the next run of `/spec:refine` knows which items have been handled.

4. **Save after each update** (save-as-you-go approach)

5. **Confirm each update:**
   ```
   ✅ Updated: {brief description of change}
   ```

### Step 6: Summary and Next Steps

After all issues in this session have been addressed:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Refinement Session Complete
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**Specification:** doc/specs/{slug}/02-specification.md
**Issues Resolved This Session:** {count}

### Changes Made

1. {Brief description of change 1}
2. {Brief description of change 2}
...

### Status

{If more issues likely remain or validation should be re-run:}

The specification has been updated. Run `/spec:refine` again to re-validate
and address any remaining issues.

{If all known issues resolved:}

All identified issues have been addressed. Run `/spec:refine` once more to
confirm the specification is ready, or proceed to:

/spec:decompose doc/specs/{slug}/02-specification.md
```

## Example Usage

```bash
# Auto-select most recently modified specification
/spec:refine

# Or specify a path explicitly
/spec:refine doc/specs/fix-chat-scroll-bug/02-specification.md
```

This will:
1. Locate the specification document
2. Run validation to identify issues
3. Present each issue interactively for resolution
4. Update the spec based on decisions
5. Summarize changes and recommend next steps

---

## Notes

- **Re-entrant:** Run multiple times until spec is ready - each run is a fresh validation pass
- **Save-as-you-go:** Each update saved immediately to prevent data loss
- **Non-destructive:** Original content preserved with strikethrough when questions resolved
- **User-driven:** All decisions made by user, not assumed
